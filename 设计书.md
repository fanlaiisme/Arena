# 智创游戏 · 作品设计书

> **作品名称**：Arena — AI博弈竞技场
>
> **赛道**：游戏创意赛道（AI + 游戏方向）
>
> **关键词**：LLM智能体博弈、不对称信息、密封拍卖、疲劳经济、多维度评估

---

## 一、摘要

**一句话概括**：Arena是一个让两个大语言模型（LLM）智能体在不对称信息下进行三天密封拍卖博弈与回合制1v1对战的实验平台——融合实时对战引擎、三方经济系统和多维度AI行为评估体系，探索LLM在不完全信息博弈中的策略涌现、推理幻觉和经济理性边界。

本项目将传统格斗游戏的物理引擎与前沿LLM的博弈推理能力相结合，构建了一套完整的"博弈实验"系统：底层是一个包含20个角色的2D实时对战游戏；中间层是一套三天赌局实验框架，涵盖暗标拍卖、疲劳经济、point转移和每日结算；顶层是Web实时可视化仪表盘和七维AI评估系统。整套系统可同时作为研究工具（观测LLM博弈行为）、教育平台（理解博弈论概念）和游戏原型（AI决策驱动的回合制竞技），实现了"AI玩游戏"到"AI在博弈中做决策"的跨越。

---

## 二、背景与需求

### 2.1 行业背景

**AI与游戏融合趋势**：近年来，大语言模型（LLM）在游戏领域展现出巨大潜力。从DeepMind的AlphaStar到Stanford的Generative Agents，AI在游戏中已从"操作执行者"进化为"策略决策者"。然而，现有研究多聚焦于AI与固定环境交互，缺乏对**多AI智能体之间博弈行为**的系统性观测平台。

**LLM博弈行为研究的缺失**：当前LLM基准测试（如MMLU、GSM8K）主要评估知识和推理能力，极少关注LLM在不完全信息博弈中的表现——包括策略一致性、经济理性、信息利用效率和对其他智能体的建模能力。这使得开发者难以预判LLM在实际博弈场景中的行为边界。

**非完全信息决策**：传统博弈论（纳什均衡、minimax）建立在完全理性和已知规则之上，而LLM智能体表现出的行为可能偏离经典博弈解。本项目构建的实验框架提供了一个观察窗口，帮助理解LLM在拍卖出价、战术部署、对手建模等任务中的实际决策过程。

### 2.2 需求分析

1. **需要一个可重复的AI博弈实验环境**：支持标准化实验流程、可配置参数、自动评估和结果复现
2. **需要观测LLM在不完全信息下的策略行为**：信息不对称（仅部分角斗士胜率已知）、对手意图不可直接观察、奖励结构与出价成本的权衡
3. **需要多维度的评估体系**：不仅看胜负，更要从规则理解、数字幻觉、策略质量、经济理性、信息利用、对手建模等多维度分析
4. **需要实时可视化的观测手段**：支持实时SSE推送、前后端分离、多页面分析

## 三、设计方案

### 3.1 整体架构

系统由三大子系统组成，呈层次化架构：

```
┌──────────────────────────────────────────────────────┐
│                   Web可视化层                         │
│  FastAPI + SSE → dashboard.html + reflections.html   │
├──────────────────────────────────────────────────────┤
│                  博弈实验层                           │
│  拍卖引擎 · 部署调度 · 经济结算 · 疲劳管理 · 评估器    │
├──────────────────────────────────────────────────────┤
│                  底层对战引擎                         │
│  Pygame 2D引擎 · 20角色 · 14武器 · 碰撞系统 · 状态效果 │
└──────────────────────────────────────────────────────┘
```

**子系统一：底层对战引擎**（`main.py`、`weapon.py`、`projectile.py`、`pet.py`、`bomb.py`、`lightning.py`）
- 800×800的2D实时对战游戏，60FPS
- 20个预设角色，每个角色拥有独特的技能组合（投射物、武器、宠物、炸弹、闪电等）
- 完整的碰撞检测矩阵和状态效果系统（减速、灼烧、电击、隐身等）
- 支持GUI人机对战和无渲染AI自动对战双模式

**子系统二：博弈实验层**（`role/main.py`、`role/auction.py`、`role/squad.py`、`role/evaluator.py`）
- 三天赌局实验流程：拍卖→部署→对战→结算→复盘，每天循环
- 三方货币经济系统（游戏币、point池、奖励池）和疲劳机制
- 七维AI评估体系，程序化检测+LLM语义评估

**子系统三：Web可视化层**（`role/web_main.py`、`role/visualizer.py`、`role/templates/dashboard.html`）
- FastAPI + SSE 实时推送，三栏仪表盘布局
- 实验过程全程可视化，支持复盘分析和反思页

#### 3.2.4 碰撞检测与状态效果

系统维护完整的碰撞矩阵，涵盖玩家-投射物、玩家-宠物、玩家-武器、玩家-炸弹、弹丸-盾牌等多种碰撞对。碰撞结果包括伤害、状态效果施加、实体销毁等。

#### 3.2.5 无渲染对战模式

通过`SDL_VIDEODRIVER=dummy`环境变量，对战引擎可在无GUI环境下运行，用于AI自动对战。每场比赛限时120秒，超时判定为平局。此模式是博弈实验层的对战基础设施。

### 3.3 博弈实验设计

博弈实验是系统的核心创新，构建了一个为期三天的完整博弈场景。

#### 3.3.1 实验参与方

两名LLM智能体玩家（默认角色：斑目貘和夜神月），各自拥有初始800游戏币，在三天内通过拍卖获得角斗士、部署出战、赢取积分，最终以游戏币总额决出胜负。

#### 3.3.2 三天实验流程

每天包含六个阶段，完整流程如下：

```
┌──────────────────────────────────────────────────────────────┐
│ Day N (N=1,2,3)                                              │
│                                                              │
│ Phase 0:   赛前情报展示                                       │
│            · 随机角斗士胜率预览（Day1:5个, Day2:4个, Day3:3个）    │
│            · Day1额外：全部20人名单 + 匿名胜率排名                 │
│            ↓                                                 │
│ Phase 0.5: 规则解读（并行）                                     │
│            每个玩家独立分析游戏机制                               │
│            ↓                                                 │
│ Phase 1:   拍卖阶段（暗标密封出价，并行）                          │
│            · 从20名中随机抽9名，逐个展示                          │
│            · 双方各自<bid>N</bid>出价，系统比价                   │
│            · 赢家支付出价获得角斗士，输家出价转入自己奖励池           │
│            · 一方满3人→另一方自动补齐（85币/人）                   │
│            · 赛后拍卖分析                                        │
│            ↓                                                 │
│ Phase 2:   部署阶段（并行）                                     │
│            玩家用<deploy slot="N">char_id</deploy>部署          │
│            ↓                                                 │
│ Phase 3:   对战阶段（3场比赛）                                   │
│            · 无渲染1v1对战，120秒限时                            │
│            · Point转移：胜者夺取min(己方point, 败方point)        │
│            · 第2局×1.5倍率                                      │
│            ↓                                                 │
│ Phase 4:   每日结算                                            │
│            · point池→奖励池（全额转移）                           │
│            · 日胜者（胜场多者）：奖励池兑换游戏币                   │
│              (≤0→0, <50→全部, ≥50→50)                         │
│            ↓                                                 │
│ Phase 5:   日末复盘（思考模式，并行）                              │
│            填写匿名排名表 + 估计对手游戏币                         │
│            ↓                                                 │
│ Phase 6:   评估（Evaluator七维评估）                             │
│            ↓                                                 │
│ Day advancement: 疲劳恢复，部署清空                              │
└──────────────────────────────────────────────────────────────┘
```

三天结束后，双方point池和奖励池按1:1兑换为游戏币，比较游戏币总额决出最终胜者。

#### 3.3.3 三货币经济系统

实验设计了三个独立但相互转化的货币/积分池，构成了策略博弈的核心：

| 货币 | 归属 | 来源 | 用途 | 流动性 |
|------|------|------|------|--------|
| **游戏币** | 角色(Role) | 初始800，每日胜者转化 | 拍卖出价，系统补齐付款 | 即时可用（"活钱"） |
| **Point池** | 阵容(Squad) | 比赛point结算（可为负） | 日末转移至奖励池 | 比赛期间绑定 |
| **奖励池** | 角色(Role) | 拍卖输家出价 + 日末point池转移 | 每日胜者兑换，最终结算 | 冻结资产（"死钱"） |

关键设计意图：
- **活钱与死钱的分离**：奖励池不能在拍卖中使用，迫使玩家在"花费游戏币竞拍"和"保护游戏币避免冻结"之间权衡
- **双方都扣出价**：暗标中无论输赢都扣除自己的出价，输方的出价转入自己的奖励池（死钱化），创造"赢者支付、输者冻结"的独特博弈结构
- **第2局×1.5倍率**：创造策略杠杆，促使玩家在部署时考虑point分配的风险收益

#### 3.3.4 疲劳系统

每个角斗士具有疲劳状态，影响出战时HP倍率：

| 疲劳等级 | HP倍率 | 说明 |
|----------|--------|------|
| 1.0（新鲜） | 100% | 未出战或充分休息 |
| 0.9 | 90% | 曾出战，已休息恢复 |
| 0.8 | 80% | 昨天出战过 |
| 0.6 | 60% | 连续两天以上出战 |

- 出战即恶化（1.0→0.8, 0.9→0.8, 0.8→0.6）
- 每日结算时未出战角斗士恢复一级
- 每天重新拍卖，阵容清空，疲劳状态跨天延续

#### 3.3.5 信息不对称设计

系统刻意制造信息不对称，是实验的核心特征：

1. **情报不对称**：双方每天随机获得不同角斗士的胜率预览（Day1:5个, Day2:4个, Day3:3个），知道的信息不同
2. **匿名胜率排名**：Day1开局展示全部20名的胜率排名但不告知对应关系，玩家需通过情报推断
3. **拍卖信息差**：暗标出价看不到对方出价金额，只在结果揭晓时知道谁赢了
4. **对手阵容可见但部署不透明**：知道对手阵容但不知道会派谁出战哪一局
5. **对手游戏币不可直接观测**：只能通过对手出价行为间接推断

### 3.4 AI智能体交互

#### 3.4.1 纯提示词驱动架构

与传统的function-calling工具调用模式不同，本实验采用**纯文本标签解析**的方式与LLM交互：

- 玩家通过`<bid>金额</bid>`标签输出拍卖出价
- 玩家通过`<deploy slot="N">char_id</deploy>`标签输出部署决策
- 使用正则表达式解析标签，最多重试3次（标签缺失时提示补充）
- 所有游戏状态信息直接嵌入提示词文本中（阵容、疲劳、拍卖进度等）

这种设计的优势是：
- **简单的交互协议**：不依赖工具调用格式，兼容任何LLM
- **可观测性强**：所有交互都是纯文本，便于日志记录和事后分析
- **减少工具调用幻觉**：避免了工具参数格式错误、重复调用等常见问题

#### 3.4.2 并行LLM调用

在可并行的阶段（拍卖出价、规则解读、日末复盘），使用`ThreadPoolExecutor`同时向两个玩家发送请求，减少总延迟。每天约产生10-20次LLM调用。

#### 3.4.3 消息历史管理

每个玩家维护独立的对话历史`message_history`，系统消息在每个阶段前追加到历史中。对话历史在阶段切换时适当截断以控制token消耗。

### 3.5 评估体系

Evaluator模块提供七维评估（E1-E5为每日评估的前五个维度，M1-M7为正式七维体系）：

| 维度 | 名称 | 实现方式 | 评估内容 |
|------|------|----------|----------|
| **M1** | 规则幻觉检测 | LLM语义比对 | 检查玩家是否误解规则（奖励池可拍卖、暗标可加价、属性克制幻觉等） |
| **M2** | 数字幻觉检测 | LLM比对+Ground Truth | 检查胜率数据虚构、花费数额不实、对手游戏币无依据断言 |
| **M3** | 策略质量 | LLM综合分析 | 拍卖策略、部署策略、第2局×1.5利用、疲劳管理、学习适应 |
| **M4** | 经济理性 | 纯程序化计算 | 花费比、弃权率、最大出价比、死钱比例、系统补齐次数、破产风险 |
| **M5** | 信息利用 | LLM评估+Ground Truth | 预览信息利用质量、拍卖信号解读、匿名排名表填写准确性 |
| **M6** | 对手建模 | LLM语义分析 | 对手拍卖行为解读、部署预测准确度、换位思考(Theory of Mind)能力 |
| **M7** | 游戏币估计 | 纯程序化正则匹配 | 从复盘文本中提取对手游戏币估计，与真实值比对 |

其中M4和M7为纯程序化评估（无LLM参与），保证评估的客观性和可重复性；其余维度结合程序化预处理和LLM语义分析。

### 3.6 Web可视化

#### 3.6.1 SSE实时推送架构

```
浏览器                           FastAPI (port 8000)
──────                           ──────────────────
dashboard.html  ──SSE──►  GET /events → Visualizer.event_stream()
                          POST /start → 后台线程 → main.run_experiment(visualizer=viz)
reflections.html ──poll──► GET /api/reflections → JSON数据
                  ──page──► GET /reflections → reflections.html
```

`Visualizer`类维护一个线程安全的`asyncio.Queue`，实验线程通过`viz.emit(event_type, data)`推送事件，FastAPI的SSE端点通过`async for event in viz.event_stream()`消费事件并推送到浏览器。

## 四、实现方式

### 4.1 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| 对战引擎 | Python 3 + Pygame | 2D实时渲染、碰撞检测、粒子效果 |
| 博弈实验 | Python 3 + ThreadPoolExecutor + asyncio | 并行LLM调用、实验流程编排 |
| AI接口 | DeepSeek API (deepseek-v4系列) | LLM推理、思考模式 |
| Web服务 | FastAPI + SSE (Server-Sent Events) | 实时数据推送、静态页面服务 |
| 前端 | 原生HTML/CSS/JavaScript | 仪表盘、反思页、事件消费 |
| 数据持久化 | JSONL + JSON | 实验日志、对战胜率统计 |

### 4.2 关键实现算法

**无渲染对战**：通过`SDL_VIDEODRIVER=dummy`和`pygame.display.set_mode`创建无窗口渲染上下文，复用完整的游戏逻辑循环（60FPS模拟），120秒超时自动判平。比赛结果（胜负、剩余HP）通过`MatchLogger`记录到JSONL文件。

**密封拍卖引擎**：`AuctionSession`管理9个角斗士的逐个拍卖流程。双方各自独立提交出价（并行LLM调用），系统收集后统一比价。支持平局重拍（最多1次）、一方满3人后自动补齐（85币/人）、低币<50强制弃权、破产即结束等边界条件。

**并发模型**：实验主线程使用`ThreadPoolExecutor`实现并行LLM调用。Visualizer使用`asyncio.Queue`实现跨线程事件通信（同步实验线程→异步FastAPI SSE），确保线程安全。

**标签解析**：使用正则表达式`<bid>\s*(-?\d+)\s*</bid>`和`<deploy\s+slot="(\d+)"\s*>\s*(\S+)\s*</deploy>`解析玩家输出，容错markdown代码块包裹。标签缺失时自动追加提示消息并重试，最多3次。

### 4.3 系统参数

| 参数 | 值 | 说明 |
|------|-----|------|
| 初始游戏币 | 800/人 | 每位玩家起始资金 |
| 拍卖起拍价 | 50 | 最低出价（0=弃权） |
| 拍卖一口价上限 | 150 | 单次出价上限 |
| 系统补齐价格 | 85/人 | 一方满3人后的自动补位价 |
| 低币阈值 | 50 | 低于此值强制弃权 |
| 每日预览数 | 5/4/3 | 按天递减的情报数量 |
| 每日比赛数 | 3局 | 第2局×1.5倍率 |
| 比赛限时 | 120秒 | 超时判平 |
| 每日胜者兑换 | ≤0:0, <50:全部, ≥50:50 | 奖励池→游戏币转换规则 |
| 疲劳HP倍率 | 100%/90%/80%/60% | 四级疲劳梯度 |

## 五、创新点

### 创新点一：纯提示词驱动的博弈智能体

区别于传统的tool-calling/function-calling智能体架构，本系统采用纯文本标签解析与LLM交互。智能体通过自然语言理解游戏规则、分析局势、输出`<bid>`/`<deploy>`等XML标签完成决策。这种设计的优势在于：(1) 零工具依赖，兼容性广；(2) 交互全文本化，便于日志审计和行为分析；(3) 更接近人类玩家的交互方式，博弈行为更具研究价值。

### 创新点二：多维度七维AI评估体系

传统AI博弈实验只关注"谁赢了"，本系统构建了七维评估框架（规则幻觉M1、数字幻觉M2、策略质量M3、经济理性M4、信息利用M5、对手建模M6、游戏币估计M7），其中M4和M7为纯程序化计算（无需LLM参与），保证客观可重复。这套评估体系不仅评判游戏结果，更从认知能力、经济理性、信息推理和社交建模等维度剖析AI智能体的博弈表现，为LLM行为研究提供了细粒度的观测工具。

### 创新点三：信息不对称下的密封拍卖博弈

实验刻意构建了多层信息不对称：(1) 情报不对称——双方了解不同角斗士的胜率；(2) 出价不透明——暗标一次性出价，看不到对方出价过程；(3) 经济结构不对称——输方出价从"活钱"变为"死钱"，创造流动性陷阱。同时设计了"双方都扣出价"的暗标规则——无论输赢都支付，输者出价冻结为死钱——这一独特机制迫使玩家在"竞拍支出"和"避免现金流冻结"间进行非平凡权衡，超越了传统的英式拍卖或维克里拍卖的博弈复杂度。

### 创新点四："对战引擎×博弈框架×可视化"的三层解耦架构

系统将实时对战物理引擎、博弈实验逻辑和Web可视化三层彻底解耦。底层对战引擎可独立运行GUI模式或供人类玩家操作；中间博弈层通过无渲染调用对战引擎，完全独立于UI；顶层Web可视化通过SSE事件队列消费博弈层事件。这种架构使得各层可独立开发、测试和复用——同一套博弈逻辑可接入不同的对战引擎，同一套可视化可观测不同的实验配置。

### 创新点五：人机双模式的实验兼容性

系统同时支持两种运行模式：(1) 人类玩家通过Pygame GUI直接操作角色进行对战；(2) AI智能体通过文本标签驱动拍卖+部署决策，再由无渲染对战引擎执行比赛。这使得系统既可以作为游戏产品（人机对战/双人对战），也可以作为研究工具（AI博弈实验），实现了娱乐性与科研价值的统一。同时，人类玩家模式为该实验提供了重要的"人类基线"——未来可对比人类与LLM在相同博弈规则下的决策差异。

### 创新点六：实时可视化的博弈观测系统

通过SSE（Server-Sent Events）实现实验过程的毫秒级实时推送，前端三栏仪表盘同步展示双方经济状态、拍卖进程、部署决策和比赛结果。独立的反思分析页面支持实验后的多维度复盘。这种"实验即演示"的设计理念，使得博弈实验过程本身就可作为教学演示和成果展示的素材，降低了博弈实验的理解门槛。

## 六、总结与展望

Arena——AI博弈竞技场，是一个将实时对战游戏与LLM博弈智能体深度融合的创新实验平台。它不是一个简单的"AI玩游戏"demo，而是一个系统性地探索LLM在不完全信息博弈中的策略涌现、认知偏差和经济理性的研究工具。

**当前成果**：20角色2D对战引擎、三天博弈实验框架、三货币经济系统、疲劳机制、七维AI评估体系、SSE实时可视化仪表盘。

**未来方向**：(1) 引入人类玩家基线对比组——将人类在相同博弈规则下的决策与LLM对比；(2) 支持更多LLM提供商对比实验（GPT/Claude/Gemini等）；(3) 增加合作博弈模式——探索两个AI在混合动机场景下的合作与背叛；(4) 对角斗士能力进行自动平衡校准——基于大量对战数据的胜率统计进行数值调整；(5) 扩展为多人在线博弈平台——支持更多玩家同时参与拍卖和部署。
